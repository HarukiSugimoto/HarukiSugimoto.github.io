第1章: インフラの全体像 — なぜインフラが必要なのか

1.1 「インフラ」とは

ソフトウェアが動くための 基盤 のこと。具体的には：

レイヤー 例 役割
ネットワーク VPC, サブネット, ルーティング 通信経路の確保
コンピュート EC2, ECS, Lambda アプリケーションの実行
ストレージ S3, EBS, EFS データの保存
データベース RDS, DynamoDB 構造化データの管理
CDN/エッジ CloudFront 高速配信・キャッシュ
セキュリティ IAM, SG, WAF アクセス制御
CI/CD GitHub Actions, CodePipeline 自動デプロイ

1.2 なぜ Infrastructure as Code (IaC) が必要か

手動構築の問題点:

  • 手作業のミス（ポートを開け忘れた、SGのルールが違うなど）
  • 再現性がない（「前と同じ設定で作って」が難しい）
  • 変更履歴が追えない
  • 環境差異（開発と本番で設定が違う）

IaCで解決されること:

  • コードレビュー可能 → インフラの変更もPRでレビューできる
  • バージョン管理 → Git で変更履歴を追跡
  • 再現性 → 同じコードで何度でも同じ環境を構築
  • 自動化 → CI/CD でインフラの変更もデプロイ

1.3 このプロジェクトのアーキテクチャ概要

ユーザー (ブラウザ)

    ▼
┌─────────────────────────────────────────────────┐
│  CloudFront (CDN)                               │
│  ├─ CloudFront Functions (IP制限)               │
│  ├─ /       → S3 (React SPA)                   │
│  └─ /api/*  → ALB → ECS Fargate (backend)      │
│              X-Origin-Verify ヘッダで検証        │
└─────────────────────────────────────────────────┘
                    │
        ┌───────────┼────────────┐
        ▼           ▼            ▼
   ┌────────┐  ┌────────┐  ┌─────────────────┐
   │  RDS   │  │   S3   │  │  FinalCode      │
   │ MySQL  │  │ Buckets│  │  Server          │
   │        │  │        │  │  (Private APIGW) │
   └────────┘  └────────┘  └─────────────────┘
                                │
                          ┌─────┴──────┐
                          │ NLB → ECS  │
                          │ (private)  │
                          └────────────┘

なぜこの構成なのか？

  • CloudFront を前面に置く理由: グローバルキャッシュ、DDoS緩和、IP制限、SSL終端を一箇所で処理
  • ALB + ECS Fargate の理由: コンテナベースでスケーラブル、サーバー管理不要
  • Fargate Spot の理由: 開発環境では最大70%コスト削減
  • FinalCode を Private にする理由: セキュリティ要件（暗号化サーバーは外部アクセス不可）
  • S3 を複数バケットに分ける理由: 用途別のアクセス制御（raw/encrypted/vcf）

第2章: Terraform とは何か — ゼロから理解する

2.1 Terraform の正体

Terraform = HashiCorp が開発した IaC ツール。

あなたが書く .tf ファイル
        │
        ▼
   Terraform CLI (terraform plan/apply)
        │
        ▼
   AWS API を呼び出してリソースを作成/更新/削除

Terraform がやること:

  1. .tf ファイルを読み取る（宣言的 = 「こうなっていてほしい」を書く）
  2. 現在の状態（State）と比較する
  3. 差分を計算して実行計画（Plan）を出す
  4. 承認後、AWS API を呼んで実際にリソースを操作する

2.2 他のツールとの比較

ツール タイプ 特徴
Terraform 宣言的 マルチクラウド対応、状態管理あり
CloudFormation 宣言的 AWS専用、YAML/JSON
Ansible 手続き的 サーバー構成管理に強い
Pulumi 宣言的 汎用プログラミング言語で書ける
AWS CDK 宣言的 TypeScript/Python でCloudFormation生成

Terraform を選ぶ理由:

  • AWS以外（GCP, Azure）にも対応
  • コミュニティが大きく、学習リソースが豊富
  • HCL（HashiCorp Configuration Language）が読みやすい
  • plan で事前に変更内容を確認できる安心感

2.3 Terraform の核心概念

┌─────────────────────────────────────────────────────┐
│                   Terraform の動作原理               │
│                                                     │
│   .tf ファイル ──→ terraform plan ──→ 差分表示      │
│       (理想)         (比較)          (確認)         │
│                       │                             │
│                       ▼                             │
│   State ファイル ←── terraform apply ──→ AWS API    │
│     (現在の状態)      (実行)            (実際の操作) │
└─────────────────────────────────────────────────────┘

State（状態ファイル）が最も重要な概念:

  • Terraform は .tf ファイルと State を比較して差分を算出する
  • State = 「Terraform が管理しているリソースの現在の状態」を記録したJSON
  • State がないと、Terraform はどのリソースを管理しているか分からない
  • このプロジェクトでは S3 (digital-arts-dev-tfstate) に保存している

2.4 Terraform のワークフロー

# 1. 初期化: プロバイダー（AWS等）のプラグインをダウンロード
terraform init

# 2. 計画: 何が変わるかを事前確認（実際には何もしない）
terraform plan

# 3. 適用: 実際にAWSリソースを作成/変更
terraform apply

# 4. 破棄: 全リソースを削除（本番では絶対にやらない！）
terraform destroy

第3章: Terraform の書き方 — 基本文法

3.1 ファイル構成の基本

Terraform は 同じディレクトリの .tf ファイルを全て読み込む。ファイル名に特別な意味はないが、慣習がある：

ファイル名 役割
main.tf メインのリソース定義
variables.tf 入力変数の宣言
outputs.tf 出力値の定義
providers.tf プロバイダー設定（AWS等）とバックエンド設定
terraform.tfvars 変数の値（環境固有の値）
locals.tf ローカル変数（計算結果のキャッシュ等）
data.tf データソース（既存リソースの参照）
versions.tf Terraform とプロバイダーのバージョン制約

3.2 Provider（プロバイダー）

「どのクラウドを使うか」を宣言する。

# providers.tf — このプロジェクトの実際のコード
terraform {
  required_version = ">= 1.6.0"

  required_providers {
    aws = {
      source  = "hashicorp/aws"    # AWS プロバイダーを使う
      version = "6.28.0"           # バージョン固定
    }
  }

  # State の保存先
  backend "s3" {
    bucket = "digital-arts-dev-tfstate"
    region = "ap-northeast-1"
    key    = "develop/network/terraform.tfstate"
  }
}

provider "aws" {
  region = var.aws_region    # 変数で東京リージョンを指定

  default_tags {
    tags = {
      Project     = var.project       # 全リソースに自動タグ付け
      Environment = var.environment
    }
  }
}

解説:

  • required_version: Terraform CLI のバージョン制約
  • required_providers: 使用するプロバイダーとバージョン
  • backend "s3": State ファイルを S3 に保存（チーム共有のため）
  • default_tags: 全リソースに自動でタグを付与（コスト管理やリソース識別に必須）

3.3 Resource（リソース）

最も基本的な構文 — AWS リソースを作る。

resource "リソースタイプ" "ローカル名" {
  属性1 = "値1"
  属性2 = "値2"
}

実例: VPC を作る

resource "aws_vpc" "main" {
  cidr_block           = "10.20.0.0/16"   # IPアドレス範囲
  enable_dns_support   = true              # DNS有効化
  enable_dns_hostnames = true

  tags = {
    Name = "digital-arts-develop-vpc"
  }
}

実例: サブネットを作る（VPC に依存）

resource "aws_subnet" "public" {
  vpc_id                  = aws_vpc.main.id    # 上の VPC を参照！
  cidr_block              = "10.20.0.0/24"
  availability_zone       = "ap-northeast-1a"
  map_public_ip_on_launch = true

  tags = {
    Name = "digital-arts-develop-public-1a"
  }
}

ポイント: aws_vpc.main.id という形で 他のリソースの属性を参照 できる。 Terraform はこの参照関係から自動的に 依存関係を解決 し、VPC → サブネット の順で作成する。

3.4 Variable（変数）

外部から値を注入するための仕組み。

# variables.tf — 変数の宣言
variable "aws_region" {
  description = "AWS リージョン"
  type        = string
  default     = "ap-northeast-1"
}

variable "vpc_cidr" {
  description = "VPC の CIDR ブロック"
  type        = string
  # default がないので、必ず指定が必要
}

variable "public_subnets" {
  description = "パブリックサブネットの CIDR リスト"
  type        = list(string)    # 文字列のリスト
}

variable "enable_nat_gateway" {
  description = "NAT Gateway を作成するかどうか"
  type        = bool
  default     = false
}

variable "tags" {
  description = "追加タグのマップ"
  type        = map(string)
  default     = {}
}

変数の型一覧:

型 例 説明
string "hello" 文字列
number 42 数値
bool true 真偽値
list(string) ["a", "b"] 文字列のリスト
map(string) {key = "val"} 文字列のマップ
object({...}) 構造体 複合型

変数の値を渡す方法（優先順位順）:

# 1. terraform.tfvars（最も一般的）
vpc_cidr = "10.20.0.0/16"
public_subnets = ["10.20.0.0/24", "10.20.1.0/24"]
enable_nat_gateway = true

# 2. コマンドラインフラグ
# terraform apply -var="vpc_cidr=10.20.0.0/16"

# 3. 環境変数
# export TF_VAR_vpc_cidr="10.20.0.0/16"

# 4. default 値（variables.tf 内）

3.5 Output（出力）

他のスタックやユーザーに値を公開する。

# outputs.tf
output "vpc_id" {
  description = "作成した VPC の ID"
  value       = module.network.vpc_id
}

output "alb_origin_header_value" {
  description = "CloudFront → ALB 検証用ヘッダの値"
  value       = random_password.origin_verify.result
  sensitive   = true    # terraform output で隠す
}

重要: sensitive = true にすると、terraform output でマスクされる。シークレットに使う。

3.6 Data Source（データソース）

既存の リソースや外部情報を参照する（作成はしない）。

# 現在のAWSリージョンの情報を取得
data "aws_region" "current" {}

# 現在のAWSアカウントIDを取得
data "aws_caller_identity" "current" {}

# 利用可能なAZを取得
data "aws_availability_zones" "available" {
  state = "available"
}

# CloudFrontのマネージドプレフィックスリスト（IPリスト）を取得
data "aws_ec2_managed_prefix_list" "cloudfront" {
  name = "com.amazonaws.global.cloudfront.origin-facing"
}

# 使い方
resource "aws_security_group_rule" "alb_from_cloudfront" {
  # CloudFront の IP レンジからのみ ALB へのアクセスを許可
  prefix_list_ids = [data.aws_ec2_managed_prefix_list.cloudfront.id]
}

3.7 Local（ローカル変数）

計算結果やよく使う値をキャッシュする。

3.8 count と for_each — 繰り返し

count: 同じリソースを N 個作る

# 2つのパブリックサブネットを作る
resource "aws_subnet" "public" {
  count = length(var.public_subnets)    # リストの長さ分ループ

  vpc_id            = aws_vpc.main.id
  cidr_block        = var.public_subnets[count.index]    # 0, 1, ...
  availability_zone = data.aws_availability_zones.available.names[count.index]

  tags = {
    Name = "${local.prefix}-public-${count.index}"
  }
}

# 参照: aws_subnet.public[0].id, aws_subnet.public[1].id
# 全IDのリスト: aws_subnet.public[*].id

for_each: マップやセットでループ

# 複数のS3バケットを作る
variable "buckets" {
  default = {
    raw       = { versioning = true }
    encrypted = { versioning = true }
    vcf       = { versioning = false }
  }
}

resource "aws_s3_bucket" "this" {
  for_each = var.buckets
  bucket   = "${local.prefix}-${each.key}"    # each.key = "raw", "encrypted", "vcf"
}

# 参照: aws_s3_bucket.this["raw"].id

使い分け:

  • count: 単純な数のループ or 条件分岐（count = var.enabled ? 1 : 0）
  • for_each: 名前付きリソースの管理（追加・削除時に他のリソースに影響しない）

3.9 条件分岐

# count を使った条件分岐（最も一般的）
resource "aws_nat_gateway" "main" {
  count = var.enable_nat_gateway ? 1 : 0    # true なら1個、false なら0個

  allocation_id = aws_eip.nat[0].id
  subnet_id     = aws_subnet.public[0].id

# 三項演算子
locals {
  instance_type = var.environment == "production" ? "db.r6g.large" : "db.t4g.micro"

3.10 テンプレートファイル

# templatefile() 関数で外部ファイルにTerraform変数を埋め込む
resource "aws_cloudfront_function" "ip_allowlist" {
  name    = "${local.prefix}-ip-allowlist"
  runtime = "cloudfront-js-2.0"

  # JavaScript テンプレートに変数を注入
  code = templatefile("${path.module}/function.js.tftpl", {
    allowed_ips   = jsonencode({ for ip in var.allowed_ips : ip => true })
    allowed_cidrs = jsonencode(var.allowed_cidrs)
  })
}

実際のテンプレート (function.js.tftpl) の中では ${allowed_ips} のように参照される。 このプロジェクトでは CloudFront Functions の IP 制限ロジックをこの方法で動的生成している。

3.11 Remote State（リモートステート参照）

異なるスタック間でデータを共有する仕組み。このプロジェクトの最重要パターン。

# stacks/services/data.tf
# → network スタックの出力値を参照する
data "terraform_remote_state" "network" {
  backend = "s3"
  config = {
    bucket = "digital-arts-dev-tfstate"
    key    = "develop/network/terraform.tfstate"
    region = "ap-northeast-1"
  }
}

# 使い方
resource "aws_lb" "backend" {
  subnets = data.terraform_remote_state.network.outputs.public_subnet_ids
  security_groups = [
    data.terraform_remote_state.network.outputs.alb_sg_id
  ]
}

なぜ Remote State が必要か？

  • スタックを分割すると、各スタックは独立した State を持つ
  • network で作った VPC ID を services で使いたい → Remote State で参照
  • 疎結合にしつつ、必要な値だけ共有できる

4.2 なぜスタック分割するのか？

1つの巨大な main.tf にしない理由:

問題 スタック分割で解決
apply が遅い（全リソース計算） 変更したスタックだけ apply
影響範囲が大きい ネットワーク変更がDBに影響しない
チームでの競合 担当スタックだけ触る
障害の局所化 edge が壊れても data は無事

分割の基準 = 変更頻度 と 影響範囲:

  • network: めったに変わらない（VPC は最初に作ったら基本そのまま）
  • services: アプリのデプロイ設定変更で頻繁に触る
  • edge: CloudFront の IP 制限追加で時々変更

4.3 依存関係チェーン（apply 順序）

stacks/network ─────────────────────────────────────┐
    │                                                │
    ▼                                                │
stacks/data ────────────────────────────┐            │
    │                                   │            │
    ▼                                   ▼            │
stacks/services ──────────► stacks/finalcode         │
    │                           │                    │
    ▼                           │                    │
stacks/edge ◄───────────────────┘                    │
    │                                                │
stacks/sftp ◄───────── stacks/data                   │
stacks/build ◄──────── stacks/edge (OIDC provider)   │

なぜこの順序か？

  1. network が最初: 全てのリソースは VPC 内に配置される
  2. data が次: RDS はサブネットとSGが必要
  3. services が次: ALB はサブネットが必要、ECS は RDS エンドポイントが必要
  4. edge が最後: CloudFront は ALB の DNS 名が必要

第5章: 各スタックの詳細解説

5.1 stacks/network — ネットワーク基盤

作るもの: VPC, サブネット, NAT Gateway, セキュリティグループ

VPC: 10.20.0.0/16 (65,536 IPs)
├── Public Subnets (インターネットアクセス可能)
│   ├── 10.20.0.0/24  (ap-northeast-1a) ← ALB, NAT Gateway
│   └── 10.20.1.0/24  (ap-northeast-1c) ← ALB (冗長性)
│
├── Private Subnets (インターネットから直接アクセス不可)
│   ├── 10.20.10.0/24 (ap-northeast-1a) ← ECS, RDS
│   └── 10.20.11.0/24 (ap-northeast-1c) ← ECS, RDS (冗長性)
│
├── Internet Gateway ← Public Subnet のインターネット出口
└── NAT Gateway ← Private Subnet からの外向き通信

なぜ Public/Private に分けるか？

  • Public Subnet: ALB など、インターネットからアクセスが必要なもの
  • Private Subnet: RDS, ECS など、外部から直接アクセスさせたくないもの
  • NAT Gateway 経由で Private → インターネットへの通信は可能（パッケージダウンロード等）

セキュリティグループ（ファイアウォール）設計:

[CloudFront IP] ──→ [ALB SG: HTTP/80許可]
                         │
                    [Backend SG: Port 8000許可（ALB SGからのみ）]
                         │
                    [RDS SG: Port 3306許可（Backend SGからのみ）]

これが 多層防御 — 各レイヤーで許可する通信を最小限に制限している。

5.3 stacks/services — アプリケーション層

作るもの: ALB, ECS Cluster, ECS Service (backend), ECR, バッチジョブ

ALB（Application Load Balancer）の設計:

# デフォルトアクション: 403 Forbidden（直接アクセスを拒否）
default_action {
  type = "fixed-response"

5.4 stacks/edge — CDN + フロントエンド配信

作るもの: CloudFront, S3 (frontend), GitHub Actions OIDC

CloudFront 構成:

CloudFront Distribution
├── Origin 1: S3 (frontend)
│   └── OAC (Origin Access Control) でアクセス制御
│
├── Origin 2: ALB (backend API)
│   └── Custom Header (X-Origin-Verify) 付与
│
├── Behavior: / → S3 (SPA)
├── Behavior: /api/* → ALB
│
├── CloudFront Functions (Viewer Request)
│   └── IP Allowlist (許可IPからのみアクセス)
│
├── Error Pages: 403, 404 → /index.html (SPA ルーティング)
│
└── Price Class: PriceClass_200 (北米+欧州+アジア)

IP制限の仕組み（CloudFront Functions）:

// function.js.tftpl — Terraform がIPリストを注入
var ALLOWED_IPS = {"121.2.39.118": true, "39.110.206.140": true, ...};
var ALLOWED_RANGES = buildRanges(["182.248.130.128/26", ...]);

function handler(event) {
  if (isAllowed(event.viewer.ip)) {
    return event.request;     // 許可 → 通常処理
  }
  return { statusCode: 403 }; // 拒否
}

なぜ CloudFront Functions で IP 制限？

  • WAF より安い（WAF は月額固定 + リクエスト課金）
  • CloudFront Functions は無料枠が大きい
  • JavaScript で柔軟なロジックが書ける
  • Terraform の templatefile() で IP リストを動的注入

GitHub Actions OIDC:

GitHub Actions ワークフロー
    │ OIDC トークン発行
    ▼
AWS IAM (OIDC Provider 信頼)
    │ 一時認証情報を発行
    ▼
├── gha-frontend-deploy ロール → S3 にデプロイ
└── gha-backend-deploy ロール  → ECR にプッシュ + ECS 更新

なぜ OIDC？

  • AWS アクセスキーを GitHub Secrets に保存しなくていい
  • 一時認証情報で安全
  • ブランチ制限（master からのみデプロイ可能）

5.5 stacks/finalcode — FinalCode サーバー

最もセキュリティが厳しいコンポーネント。

[Backend ECS] → [VPC Endpoint] → [API Gateway (PRIVATE)] → [NLB] → [FinalCode ECS]

なぜこんな複雑な構成か？

  1. FinalCode はファイル暗号化サーバー → 外部からアクセスさせてはならない
  2. API Gateway を PRIVATE にすることで、VPC 内からしか到達不可
  3. NLB (Network Load Balancer) を internal にして、Private Subnet 内のみ
  4. VPC Endpoint でインターネットを経由せずに API Gateway に到達
Public Internet ──X──→ API Gateway (PRIVATE)  ← アクセス不可
VPC 内 Backend   ──→ VPC Endpoint ──→ API Gateway ──→ NLB ──→ FinalCode  ← OK

6.4 モジュール設計のコツ

良い設計:
├── 1つのモジュール = 1つの責務
├── 入力は明示的（暗黙の依存を避ける）
├── 出力は必要最小限
└── 環境固有の値はモジュール内に書かない

悪い設計:
├── 1つのモジュールに全部入り
├── ハードコードされた値がモジュール内に
├── モジュール同士が直接参照し合う
└── 出力が多すぎて何が重要か分からない

第7章: セキュリティ設計 — なぜこうなっているのか

7.1 多層防御 (Defense in Depth)

Layer 1: CloudFront Functions ← IP制限（許可IPのみ通過）
    │
Layer 2: CloudFront           ← DDoS緩和（AWS Shield Standard）
    │
Layer 3: ALB                  ← X-Origin-Verify ヘッダ検証
    │
Layer 4: Security Group       ← ポート + ソース制限
    │
Layer 5: Private Subnet       ← インターネットから直接到達不可
    │
Layer 6: IAM Role             ← 最小権限の原則
    │
Layer 7: Secrets Manager      ← シークレットの安全な管理

7.2 ネットワーク分離

Internet
    │
    ▼
[Public Subnet]
├── ALB (HTTP/80)           ← CloudFront からのみ
├── NAT Gateway             ← Private → Internet の出口
│
[Private Subnet]
├── ECS Fargate (backend)   ← ALB からのみ
├── ECS Fargate (finalcode) ← NLB からのみ
├── RDS MySQL               ← Backend からのみ
├── NLB (internal)          ← API Gateway からのみ
└── VPC Endpoint            ← API Gateway への入口

7.3 セキュリティグループの最小権限

# ALB: CloudFront の IP レンジからのみ
resource "aws_security_group_rule" "alb_http" {
  type              = "ingress"
  from_port         = 80
  to_port           = 80
  protocol          = "tcp"
  prefix_list_ids   = [data.aws_ec2_managed_prefix_list.cloudfront.id]
  security_group_id = aws_security_group.alb.id
}

# Backend: ALB からのみ
resource "aws_security_group_rule" "backend_from_alb" {
  type                     = "ingress"
  from_port                = var.backend_container_port
  to_port                  = var.backend_container_port
  protocol                 = "tcp"
  source_security_group_id = aws_security_group.alb.id      # ALB の SG を指定
  security_group_id        = aws_security_group.backend.id
}

# RDS: Backend からのみ
resource "aws_security_group_rule" "rds_from_backend" {
  type                     = "ingress"
  from_port                = 3306
  to_port                  = 3306
  protocol                 = "tcp"
  source_security_group_id = aws_security_group.backend.id   # Backend の SG を指定
  security_group_id        = aws_security_group.rds.id
}

チェーン構造: CloudFront IP → ALB SG → Backend SG → RDS SG 各 SG は 直前のレイヤーからのみ アクセスを許可。

7.4 IAM ロール設計

ECS Task Execution Role (Terraform が管理)
├── ecr:GetAuthorizationToken     ← ECR からイメージ取得
├── ecr:BatchGetImage
├── logs:CreateLogStream          ← CloudWatch Logs 書き込み
├── logs:PutLogEvents
├── secretsmanager:GetSecretValue ← シークレット取得（特定ARNのみ）
└── kms:Decrypt                   ← 暗号化キー復号

ECS Task Role (アプリケーションが使う権限)
├── s3:GetObject (raw-files)      ← ファイル読み取り
├── s3:PutObject (raw-files)      ← ファイルアップロード
├── s3:GetObject (encrypted-files)← 暗号化ファイル読み取り
└── execute-api:Invoke            ← FinalCode API Gateway 呼び出し

Execution Role vs Task Role の違い:

  • Execution Role: ECS の基盤が使う（コンテナ起動前の準備）
  • Task Role: コンテナ内のアプリが使う（実行時の権限）

    Action   = "s3:*"           # 権限が広すぎる
    Resource = "*"              # 対象が広すぎる
  }]
})

policy = jsonencode({
  Statement = [{
    Effect   = "Allow"
    Action   = ["s3:GetObject", "s3:PutObject"]   # 必要な操作のみ
    Resource = "${aws_s3_bucket.raw.arn}/*"         # 特定バケットのみ
  }]
})

# ❌ depends_on の乱用
resource "aws_ecs_service" "backend" {
  depends_on = [aws_lb.backend]    # 暗黙の依存で十分な場合は不要
  # load_balancer ブロックで ALB を参照していれば自動的に依存する
}

第11章: トラブルシューティング

11.1 よくあるエラーと対処法

1. State Lock エラー

Error: Error acquiring the state lock

→ 他の人が terraform apply 中。終わるまで待つか、手動ロック解除:

terraform force-unlock <LOCK_ID>

2. リソースが既に存在する

Error: Creating S3 Bucket: BucketAlreadyExists

→ 手動作成されたリソースを import:

terraform import aws_s3_bucket.this my-existing-bucket

3. 依存関係エラー

Error: Reference to undeclared resource

→ data "terraform_remote_state" が正しく設定されているか確認。 依存元のスタックが先に apply されているか確認。

4. Provider バージョンの不一致

Error: Failed to query available provider packages

→ terraform init -upgrade でプロバイダーを更新

5. 権限エラー

Error: AccessDenied

→ AWS CLI の認証情報を確認: aws sts get-caller-identity

11.2 State の復旧

# State のバックアップ確認（S3 バージョニングが有効なら）
aws s3api list-object-versions \
  --bucket digital-arts-dev-tfstate \
  --prefix develop/network/terraform.tfstate

# 特定バージョンのダウンロード

11.3 安全な変更の進め方

# 1. 必ず plan を先に見る
terraform plan

# 2. destroy が含まれていないか確認
# "will be destroyed" が出たら要注意！

# 3. 変更が少ないことを確認
# Plan: 1 to add, 0 to change, 0 to destroy.  ← 安心
# Plan: 0 to add, 0 to change, 15 to destroy. ← 危険！確認が必要

# 4. 不安なら target で1つずつ
terraform apply -target=aws_security_group_rule.new_rule
